{% docs dim_listings_w_hosts__updated_at %}
Timestamp of the most recent change to either the listing or the host profile,
whichever is later.

Computed as:

```sql
GREATEST(listing.updated_at, host.updated_at)
```

This means the column reflects the last time *anything meaningful changed*
in the combined record — whether that was the host updating their profile,
the listing changing its price, or the host gaining or losing Superhost status.

Use this field when you need to detect recently changed listings-with-hosts
records, or when filtering for records that have been updated since a
given date. Do not use it to determine when specifically the listing changed
vs when the host changed — for that, join directly to `dim_listings_cleansed`
or `dim_hosts_cleansed` and compare their individual `updated_at` values.
{% enddocs %}


{% docs fct_reviews__review_id %}
Surrogate primary key that uniquely identifies each review.

Generated by hashing the concatenation of four fields:

| Field | Role |
|---|---|
| `listing_id` | Which listing was reviewed |
| `review_date` | When the review was written |
| `reviewer_name` | Who wrote the review |
| `review_text` | What they wrote |

The hash is produced by `dbt_utils.generate_surrogate_key` (MD5 after
coalescing each component to an empty string), called via the project's
`generate_surrogate_key` macro wrapper.

This composite key is necessary because the source data has no natural
primary key on reviews — Airbnb does not expose a review ID in the raw
export. The four fields together uniquely identify a review in practice,
since a guest cannot submit two different reviews for the same listing on
the same date under the same name.

The surrogate key also enables safe **incremental loading**: because the
key is deterministic and based on content, re-processing a review that
already exists produces the same key, allowing deduplication logic to
identify and skip already-loaded records.
{% enddocs %}


{% docs mart_fullmoon_reviews__is_full_moon %}
Indicates whether the review was written on the day immediately following
a full moon.

**Values:** `'full moon'` or `'not full moon'`

**Join logic:**

```sql
LEFT JOIN seed_full_moon_dates fm
  ON TO_DATE(r.review_date) = DATEADD(DAY, 1, fm.full_moon_date)
```

The one-day offset is intentional. The hypothesis being tested is that
guests who *experienced* their stay during a full moon are more likely to
write a review the following morning. A review dated exactly on a full moon
date reflects a stay that ended that night — the guest would more likely
have been affected by the preceding night's full moon. The offset captures
this one-day lag between the lunar event and the review behaviour.

Reviews that do not match any full moon date + 1 day receive `'not full moon'`
via the LEFT JOIN's CASE WHEN NULL check. Since this is a LEFT JOIN, every
review from `fct_reviews` is preserved — the mart has exactly the same row
count as the fact table.
{% enddocs %}
